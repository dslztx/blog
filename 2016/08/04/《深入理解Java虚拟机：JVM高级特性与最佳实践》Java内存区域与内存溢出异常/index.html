《深入理解Java虚拟机：JVM高级特性与最佳实践》Java内存区域与内存溢出异常

发表于 2016-08-04 专题于 语言 ， Java ， JVM Disqus：

一、前言

1.1、进程使用内存上限

进程所能使用内存上限受到硬件和操作系统限制，比如32位Windows操作系统下进程所能使用内存上限为2G。

1.2、虚拟机进程

我们平时说的Java进程其实是虚拟机进程。通过执行“java”命令触发运行虚拟机进程，比如
“java com.dslztx.Main”和“java -version”等。

1.3、虚拟机进程内存管理

如“1.1、进程使用内存上限”所述，虚拟机进程使用内存上限受到硬件和操作系统限制。根据虚拟机规范，虚拟机进程内存管理采取内存动态分配，垃圾收集，内存分类等技术。采用该种内存管理机制的优点是内存管理更加方便简单，缺点是一旦出现内存泄漏和溢出等问题，较难排查问题。

二、虚拟机进程内存管理之内存分类

分配给进程使用的一块内存区域被称为“Native Memory”（有时候也被称为“Native Heap”）
，如“1.1、进程使用内存上限”所述，“Native Memory”上限受到硬件和操作系统限制。
“Native Memory”如何被使用由进程决定，虚拟机规范人为对虚拟机进程使用的“Native Memory”进行了划分，虽然其实底层都使用了相同的方式操作和管理内存，比如都使用C/C++的“malloc”函数。
虚拟机进程内存管理对所使用的“Native Memory”的具体分类为：程序计数器，Java虚拟机栈，本地方法栈，Java堆，方法区，直接内存，自己写的JNI方法直接所分配内存，虚拟机进程运行自身需要内存和其他使用内存。如图1所示。

另外，“运行时常量池（Runtime Constant Pool）”是以上“方法区（Method Area）”的一部分，它用来存储类加载后类的字面量和符号引用。

三、虚拟机进程内存管理之内存使用

虚拟机进程对内存的使用包括3方面内容：对象内存分配，对象内存布局和对象内存访问。由于不同虚拟机和内存类别，采用的内存使用规范不尽相同，因此，接下来限定于“HotSpot虚拟机和Java堆内存类别”来进行举例说明。

3.1、对象内存分配

由以上描述可知，对象创建时对象实例所需内存在“Java堆”上分配。分配内存有两种方式：“指针碰撞”和“空闲列表”。具体所采用的内存分配方式由“堆”是否规整决定，而“堆”的规整又由采用的垃圾收集器是否带有压缩整理功能决定。因而，在使用Serial，ParNew等带有压缩过程的垃圾收集器时，分配内存方式为“指针碰撞”；在使用CMS等不带有压缩过程的垃圾收集器时，分配内存方式为“空闲列表”。
对分配内存时的资源竞争有两种解决方案：分配内存时进行同步处理，具体是采取CAS配上失败重试的机制；在“堆”中给每个线程预先分配一块内存，称为本地线程分配缓存（TLAB），线程需要分配内存时，先在属于自身线程的TLAB中进行分配，如果失败，再同步申请新的TLAB。是否启用TLAB，通过“-XX:+/-UseTLAB”虚拟机参数决定。

3.2、对象内存布局

一个对象的内存布局分为3块区域：对象头（Header），实例数据（Instance Data）和对齐填充数据（Padding）。

3.3、对象内存访问

通过对象引用访问对象实例（即对象内存）有两种方式：使用句柄和直接指针。

3.3.1、句柄访问

Java堆中会划出一块内存用作句柄池，对象引用存储的就是句柄的地址，而句柄中包括了对象实例和类型实例（即Class对象实例）的具体地址。
优点是对象引用存储稳定的句柄地址，当对象实例被移动时，只需改变句柄中对象实例的具体地址即可；缺点是需要两次定位，才能得到对象实例的具体地址。

3.3.2、直接指针

对象引用存储对象实例的具体地址，而对象实例中又存储类型实例的具体地址。优点是只需一次定位就能得到对象实例的具体地址，相对来说速度更快；缺点是对象实例被移动时，对象引用存储的对象实例具体地址也需要相应作改变。HotSpot虚拟机采用“直接指针”方式。

4.2.4、说明

在HotSpot虚拟机中，Java虚拟机栈和本地方法栈类似，因此，这里将两者放在一起讲。
关于Java虚拟机栈/本地方法栈的StackOverflowError和OutOfMemoryError异常，书中描述得非常差劲。还是直接参考[3]中“2.5.2. Java Virtual Machine Stacks”小节内容。
接下来简要作几点阐述：
1、Java虚拟机栈/本地方法栈是线程私有的，Java虚拟机栈/本地方法栈的大小通过“-Xss”虚拟机参数设定，当线程欲使用的Java虚拟机栈/本地方法栈大小超过“-Xss”虚拟机参数设定值时，抛出StackOverflowError异常。线程内方法调用深度和方法调用产生的栈帧大小直接决定Java虚拟机栈/本地方法栈的使用总大小。
2、由于Java虚拟机栈/本地方法栈是线程私有的，因此开启N个线程，就需要申请建立N个Java虚拟机栈/本地方法栈，即需要申请N*（Xss虚拟机参数设定值）*2的内存，当这个内存申请失败时，就会抛出OutOfMemoryError异常。比如在32位Windows下，一个虚拟机进程最大可使用2GB内存，需要减掉分配给Java堆的内存，需要减掉分配给方法区的内存，程序计数器使用内存量很小，可以忽略，因此，（2GB-Java堆分配内存-方法区分配内存）是“自己写的JNI方法直接所分配内存，虚拟机进程运行自身需要内存，直接内存，其他使用内存，Java虚拟机栈，本地方法栈”所能使用内存总和的上限值。

4.3.4、说明

由以上描述可知，方法区存储已被虚拟机加载的类信息，常量，静态变量，即时编译器编译以后的代码等数据，通过CGLib库提供的字节码操作技术，我们可以在运行时动态生成大量的新类，从而使得方法区的内存不够分配，最终导致抛出OutOfMemoryError异常。
HotSpot虚拟机使用“永久代”来实现方法区，通过“-XX:PermSize=10m”和“-XX:MaxPermSize=10m”这两个虚拟机参数，可以将“永久代”的内存大小设定为固定值10MB。
使用JDK6，抛出OutOfMemoryError异常时，还会有提示“PermGen space”，表明是在使用“永久代”时抛出的异常。
